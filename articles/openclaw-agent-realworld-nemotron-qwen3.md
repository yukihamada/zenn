## マルチステップタスクの実録：ツールチェーンはどう繋がるか

最も試したかったのは「競合調査して比較表を作って」のような、複数ツールを連鎖的に使う必要があるタスクだ。

### Qwen3-32B：ちゃんと繋がるケース

```
User: 日本のノーコードツールを3つ調べて、料金と特徴を比較表にして

[Iteration 1]
  [tool_start] web_search("日本 ノーコードツール 比較 2026")
  [tool_result] web_search → 3件のURL取得 (1.2s)

[Iteration 2]
  [tool_start] read_webpage("https://example.com/nocode-comparison")
  [tool_result] read_webpage → 2400文字取得 (3.1s)

[Iteration 3]
  [thinking] 取得した情報を整理中...

| ツール | 月額 | 特徴 | 日本語対応 |
|--------|------|------|-----------|
| STUDIO | ¥1,280〜 | デザイン重視、CMS内蔵 | ◎ |
| Kintone | ¥1,500〜 | 業務アプリ、社内DB | ◎ |
| Bubble | $29〜 | 汎用、自由度高 | △ |
```

Qwen3-32Bは3イテレーションで`web_search → read_webpage → 整理・出力`のチェーンを完走した。16Kコンテキストがあるため、途中でwebページの内容を保持しながら次のステップに進める。

### Nemotron-9B：途中で脱落するケース

同じプロンプトをNemotron-9Bに投げると、違う結果になる。

```
User: 日本のノーコードツールを3つ調べて、料金と特徴を比較表にして

[Iteration 1]
  [tool_start] web_search("日本 ノーコードツール 比較")
  [tool_result] web_search → 3件のURL取得 (0.8s)

[Iteration 2]
  ← ここで read_webpage を呼ばずに直接回答を生成

以下は主なノーコードツールの比較です：
| ツール | 特徴 |
|--------|------|
| Wix | ウェブサイト作成 |
| Bubble | アプリ開発 |
...（料金情報なし、明らかにweb検索結果を使っていない）
```

これは運用中に何度も遭遇したパターンだ。Nemotron-9Bはweb_searchの結果を受け取った後、read_webpageを呼ぶべき場面で**学習データ内の知識に基づいて直接回答してしまう**。9Bパラメータではマルチステップの計画維持が難しいようで、特に2回目以降のツール呼び出しの精度が落ちる。

### 失敗の傾向まとめ

| 状況 | Nemotron-9B | Qwen3-32B |
|------|-------------|-----------|
| 単発ツール呼び出し | ほぼ成功 | ほぼ成功 |
| 2連鎖（search→read） | 5割で脱落 | 8割成功 |
| 3連鎖以上 | ほぼ脱落 | 6割成功 |
| ツール結果の要約 | 短めだが正確 | 詳細で構造的 |

## コンテキスト長が効く場面：8K vs 16K

### 会話履歴の保持

Nemotron-9B（8K）で10往復ほど会話すると、こうなる。

```
User: さっき調べてくれたノーコードツールの件だけど、Kintoneの料金もう一回教えて

Nemotron: 申し訳ございませんが、ノーコードツールについて調べた記録が
         見つかりません。改めてお調べしましょうか？
```

システムプロンプト（ツール定義+メモリコンテキスト）だけで約4,800トークンを消費する。残り3,200トークンで会話履歴を保持する必要があり、5〜6往復が限界だ。

Qwen3-32B（16K）では同じ場面で：

```
User: さっき調べてくれたノーコードツールの件だけど、Kintoneの料金もう一回教えて

Qwen3: 先ほどの調査結果によると、Kintoneは月額1,500円（1ユーザー）
       からの料金体系です。スタンダードコースとライトコースがあり...
```

16Kあれば10〜15往復の会話を保持できる。「前の話の続き」が通じるかどうかの差は、実用上かなり大きい。

### ドキュメント処理

read_webpageで取得したページが長い場合にも差が出る。Nemotron-9Bは取得結果が3,000トークンを超えるとシステムプロンプトと合わせてコンテキストが溢れ、vLLMから400エラーが返る。Lambda側でトランケートする処理を入れてはいるが、切り詰めた結果が中途半端で使い物にならないこともある。

## 完全自律の限界と現実

### 正直な実力差

同じプロンプト「Rustのasync/awaitの仕組みを初心者にわかるように説明して」で比較した。

**Nemotron-9B**：基本的な概念は説明できるが、Future traitの内部動作やPinの説明でやや不正確な記述が混じる。コード例は動くものを出すが、説明の深さが足りない。

**Qwen3-32B**：Futureの状態遷移、Wakerの仕組みまで踏み込める。ただしコード例に時々コンパイルエラーが混じる（型注釈の漏れなど）。

**Claude Opus 4.6やGPT-4o**：正確性・網羅性ともに明確に上。特に「初心者向け」の語彙選択や説明の段階構成が洗練されている。

これは事実として認める必要がある。9B・32Bモデルは、フロンティアモデルと比べると推論品質に差がある。特に以下の場面で差が顕著になる。

- 複雑な論理推論（多段階の条件分岐）
- コードの正確性（コンパイルが通るかどうか）
- 長い文章の構造化（章立てや論点整理）
- 曖昧な指示の意図汲み取り

### それでも「APIなし」で運用する価値

ではなぜ使うのか。3つの場面で自律スタックが勝つ。

**1. プライバシーが最優先の場面。** 社内の機密データを含む質問、個人の医療情報、法務関連の相談。外部APIにトークンを送信しないことの価値は、品質差では測れない。

**2. レイテンシが重要な場面。** RunPodのGPU Podは自分専用なので、API Gatewayのキュー待ちがない。Nemotron-9Bの初回トークンは200ms以下。フロンティアモデルのAPI経由では500ms〜2秒かかる。

**3. コストが線形予測可能な場面。** A5000で月$230、A6000で月$353。使い放題。トークン従量課金の予測不能性に悩まされない。月100万トークン以上使うなら、確実に安い。

## 今後の展望：WASMとエッジAI

### wasm32-wasiへのコンパイル

前回の記事で触れたRust gatewayのWASMコンパイルは、**まだ実装途中**だ。正直に書く。

現状の課題：

```
cargo build --target wasm32-wasi

error[E0433]: tokio はwasm32-wasi未対応
  → 非同期ランタイムの差し替えが必要
error: reqwest はwasm32-wasi未対応
  → HTTP clientの差し替えが必要
```

axum + tokioのフルスタックをそのままWASMにはできない。Cloudflare Workers上で動かすには、`workers-rs`クレートを使った別のエントリポイントが必要になる。gatewayのコア部分（プロトコルパース、セッション管理）は`#[cfg(target_arch = "wasm32")]`で分岐させる設計を進めている。

### Cloudflare Workers上のエージェント

成功すれば、こういう世界が来る。

```
ユーザー（世界中のどこか）
  │ HTTPS
  ▼
Cloudflare Workers Edge（ユーザー最寄りのPoP）
  │ OpenClaw Gateway（WASM, ~3MB）
  │ セッション管理、認証、ルーティング
  ▼
RunPod vLLM（東京リージョン）
  推論のみ
```

gatewayがエッジで動けば、WebSocketのハンドシェイクやセッション管理のレイテンシが劇的に改善される。LLM推論だけがGPUのある場所に集約される形だ。

### ブラウザ内LLM（WebLLM / wllama）との組み合わせ

もう一つ面白い可能性がある。ブラウザ内でWebGPUを使って小型LLMを動かすWebLLMやwllamaとの組み合わせだ。

```
ブラウザ
  ├── WebLLM（Qwen3-0.6B-GGUF, ~400MB）← 即応答・オフライン
  └── OpenClaw Gateway（WASM）
        └── RunPod vLLM（Qwen3-32B）← 高品質・オンライン
```

簡単な質問はブラウザ内の0.6Bモデルが即座に回答し、複雑なタスクだけリモートの32Bに投げる。この判定ロジックをgateway側に持たせれば、ユーザーからは単一のAIアシスタントに見える。

実装はまだ先だが、Rustで書いたgatewayがWASMとしてブラウザに載り、LLM推論もブラウザ内で完結するなら、**サーバーが一切不要な完全ローカルAIエージェント**が実現する。

## まとめ：実用ラインはどこか

| 用途 | Nemotron-9B | Qwen3-32B | フロンティア(Claude/GPT-4o) |
|------|-------------|-----------|---------------------------|
| 単発Q&A | 実用的 | 実用的 | オーバースペック |
| 単発ツール呼び出し | 実用的 | 実用的 | 実用的 |
| マルチステップタスク | 厳しい | 条件付きで可 | 実用的 |
| 長い会話の継続 | 5〜6往復が限界 | 10〜15往復 | 制限なし |
| コード生成 | 簡単なもの | 中程度まで | 複雑でもOK |

9B・32Bのセルフホストモデルは万能ではない。しかし「単発の質問応答」「1回のツール呼び出しで完結するタスク」「プライバシーが最優先の用途」では十分に実用的だ。マルチステップの複雑なタスクは、モデルサイズが上がれば改善するだろう。70B・100Bクラスがvllmで効率的に動く環境が整えば、フロンティアとの差はさらに縮まる。

完全自律は目標であって、今日の到達点ではない。だが「APIキーなしで動くAIエージェント」という選択肢が存在すること自体に価値がある。次はWASMコンパイルを完成させて、エッジまで持っていきたい。
